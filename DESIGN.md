# Evidence Note Annotator — 产品设计文档

面向材料科学论文结构化抽取结果的在线标注工具。
本工具不是 JSON 编辑器，而是一个强调 **产品逻辑、标注效率、可追溯性与质量控制** 的审核平台。

---

## 1. 产品目标与用户角色

### 1.1 产品目标
把只读的抽取流水线产物（`extracted/`）变成一个 **可审核、可修正、可补充、可确认、可追溯** 的标注工作台，产出高质量的最终标注结果与下游可消费文件。

核心价值主张：**“每一个结构化字段，都能一眼看到它从原文哪里来、质量如何、被谁改成了什么。”**

### 1.2 用户角色
| 角色 | 目标 | 关键诉求 |
|---|---|---|
| **标注员 (Annotator)** | 逐字段核对、修正、确认 | 效率（快捷键、证据自动定位）、低认知负担 |
| **复核员 (Reviewer)** | 二次检查“待复核/冲突”字段 | 差异对比、备注、状态流转 |
| **数据负责人 (Owner)** | 把关整体质量、导出交付 | 完整性检查、审计日志、可追溯导出 |

> 当前实现为单人本地部署；多人协作/权限角色见 §8 演进路线（数据模型已为其预留）。

## 2. 典型使用流程

```
选论文 → 浏览 bucket（论文级/样品级）
      → 选字段：左侧原文自动滚动 + 高亮对应证据块
      → 核对值：确认 / 修改 / 标记待复核 / 标记冲突 / 补充新字段
      → （自动保存，随时撤销/重做）
      → 完整性检查（进度条 + 未审提示）
      → 导出标注包（机器消费 + 人工审阅两类文件）
```

单字段标注循环（键盘友好）：
`j/k` 切换字段 → `c` 确认 / `x` 冲突 / `r` 待复核 → `e` 编辑值 → `n/N` 在多证据间切换。

## 3. 页面信息架构

三栏布局 + 顶栏 + 底部图例，刻意把“证据 / 字段 / 详情”三件事分离，降低认知负担。

```
┌─ 顶栏：论文切换 · 进度条 · 保存状态 · 撤销/重做 · 任务状态 · 导出 · 帮助 ─┐
├───────────────┬──────────────────────────────┬───────────────────────┤
│  左：证据原文   │  中：Buckets + 字段列表          │  右：字段详情 Inspector  │
│               │                              │                       │
│ · 证据原文/知识 │ · bucket 轨道（论文级/各样品）    │ · 路径 path            │
│   摘要 Tab     │ · 搜索 + 状态筛选               │ · 当前值（可编辑）       │
│ · 按 evidence  │ · 按 section 分组的字段行        │ · 原始值对比            │
│   block 分段    │ · 每行：状态色条 + 值 + 证据徽章  │ · 证据列表（可点定位）    │
│ · 点击字段自动   │ · 行内快捷操作（确认/复核/冲突）  │ · 支持度诊断            │
│   高亮 + 滚动   │ · “＋补充字段”                  │ · 备注                 │
├───────────────┴──────────────────────────────┴───────────────────────┤
└─ 底部：状态颜色图例 + “颜色=审核状态，徽章=证据质量” 的说明 ─────────────┘
```

### 3.1 面板联动逻辑
- **中 → 左**：点击字段 / `j`,`k` 移动 → 左侧定位并高亮该字段引用的 evidence block（多证据高亮全部、`n/N` 循环聚焦）。
- **左 → 中**：点击原文某个 block → 跳到第一个引用它的字段。
- **中 → 右**：选中字段 → 右侧展开可编辑详情与证据；编辑立即回写状态色条与进度。
- **右 → 左**：点击证据条目 → 左侧滚动聚焦该 block；改证据 ID → 重新高亮。

### 3.2 如何降低认知负担 / 避免右侧字段过多导致的混乱
- **分而治之**：右侧详情一次只服务一个字段；中间列表只是导航。
- **Bucket 收敛**：一次只看一个 bucket（论文级或单个样品），而非一次面对上百字段。
- **section 分组 + sticky 分组标题**：字段按 `papers/alloys/.../properties` 归组。
- **搜索 + 状态筛选**：可只看“待复核 / 冲突 / 缺证据 / 未处理”。
- **单一颜色维度**：颜色只编码“审核状态”，证据质量用徽章，避免颜色过载（见 §5）。

## 4. 核心功能

### 4.1 字段级 evidence_ref 跟踪与高亮
- 数据链路：`field_evidence.json.fields[].support.evidence_refs` → `verify/evidence_blocks.json` 的 `block_id`（含 `char_start/char_end`，与 `preprocess/cleaned_input.md` 精确对齐）。
- 左侧把原文渲染为 **按 evidence block 分段** 的可高亮单元 —— 高亮粒度天然等于抽取引用粒度，定位 100% 精确。
- **多证据**：全部块以浅色高亮，聚焦块以主色高亮；`n/N` 或点击证据条目在多块间切换，Inspector 显示“第 i/N”。
- **异常处理**：
  - `evidence_ref` 指向的 block 不存在 → Inspector 标 “⚠ 未找到”，左侧顶栏显示“k 块无法定位”，可手动改证据 ID 修复。
  - `evidence_refs` 为空 → 徽章“缺证据”，Inspector 明确提示“请人工核对来源”。

### 4.2 标注状态与颜色体系
见 §5。

### 4.3 Buckets 级标注
- `paper_level` bucket（论文元数据、表征方法、计算细节、未映射发现）。
- `sample_level` bucket（按 `sample_id` 分组，每个样品一个 bucket）。
- Bucket 轨道上每个 chip 显示 **该 bucket 内字段的状态分布迷你条**，一眼看出哪个样品还没标完。
- 支持为整个 bucket 设置状态/备注（数据模型 `buckets{}`，用于 bucket 级结论）。

## 5. 标注状态与颜色体系

### 5.1 状态集合（审核状态，主维度）
| 状态 | 含义 | 颜色 | 触发 |
|---|---|---|---|
| `unprocessed` 未处理 | 尚未审核 | 灰 | 默认 |
| `confirmed` 已确认 | 接受原值 | 绿 | 点“确认”/`c` |
| `modified` 已修改 | 值被编辑 | 蓝 | 编辑值（改回原值自动撤销） |
| `added` 已补充 | 人工新增字段 | 紫 | “＋补充字段” |
| `needs_review` 待复核 | 存疑，转复核 | 琥珀 | `r` |
| `conflict` 冲突 | 与原文矛盾 | 红 | `x` |

### 5.2 证据质量（次维度，用徽章而非颜色）
`证据充分 / 数值匹配 / 弱证据 / 缺证据` + 置信度数值 + `冲突` + 证据块数量。

### 5.3 避免颜色过载的规则
1. **颜色只有一个维度 = 审核状态**；证据质量、置信度、冲突全部用徽章 / 图标承载。
2. 行内只用一根左侧“状态色条” + 极浅的行底色，不整行重彩。
3. 底部常驻图例 + 一句话说明“颜色=状态，徽章=证据质量”。
4. 状态可点击切换、同状态再点即回到未处理（可逆、低负担）。

## 6. 导出：目录结构与文件清单

导出为 `<paper_id>_annotation_export.zip`：

```
<paper_id>/
├── MANIFEST.json                  [机器] 包内清单、任务状态、进度、未应用编辑列表
├── review_summary.md              [人工] 可读审核报告：变更清单 + 补充字段
├── annotation_state.json          [机器] 原始标注状态（可回灌本工具继续标）
├── field_review.json          ★   [机器] 权威产物：每字段 原值/审核值/状态/证据/置信度（扁平、完整、可靠）
├── text_extraction.reviewed.json  [机器] 尽力合并回原结构的 JSON（best-effort，未应用项见 MANIFEST）
├── diff.json                      [机器] 原值→审核值的变更集（可视化 diff / 质量分析）
└── audit_log.jsonl                [机器/人工] 追加式事件流（谁在何时把哪个字段改成了什么）
```

- **给机器消费**：`field_review.json`（★ 首选、可靠、完整）、`text_extraction.reviewed.json`（要结构时用）、`diff.json`、`annotation_state.json`、`audit_log.jsonl`。
- **给人工审阅**：`review_summary.md`、`MANIFEST.json`。
- **可追溯性**：`audit_log.jsonl` 记录每一次状态/取值/证据变更；`diff.json` 给出前后对照；`MANIFEST.unapplied_edits` 透明列出无法回填结构的字段，绝不静默出错。

> 关于路径回填：字段 `path` 是流水线自定义寻址 DSL（含带点的 id、`__` 消歧后缀、`[int]`/`[key]` 括号）。解析器覆盖了绝大多数字段；个别源数据路径本身不自洽的字段会被列入 `unapplied_edits`。无论回填成功与否，`field_review.json` 始终 100% 记录审核结果。

## 7. 专业标注平台“必须有”的能力（已实现）

| 能力 | 实现 |
|---|---|
| 任务状态管理 | 顶栏 未开始/进行中/已提交；首次操作自动置“进行中” |
| 自动保存 | 每次变更 debounce 800ms 落盘；顶栏“已保存/未保存/保存中/失败”；离开页面兜底保存 |
| 撤销 / 重做 | 客户端 100 步快照栈，`⌘Z` / `⌘⇧Z` |
| 字段搜索与筛选 | 按路径/值搜索；按状态（含“缺证据”）筛选 |
| 冲突检测 | 抽取侧 `contradiction` 徽章 + 人工“冲突”状态 |
| 完整性检查 | 进度条（已审/总数 + 各状态计数）；导出前拦截未审字段 |
| 快捷键 | `j/k c x r e n/N / ? ⌘Z ⌘⇧Z ⌘S ⌘P`（`?` 查看全表） |
| 审核流 | 状态机 未处理→确认/修改/待复核/冲突；待复核+备注支撑二次复核 |
| 历史 / 审计 | 追加式 `audit_log`，导出即审计凭证 |
| 批量视角 | bucket 迷你分布条 + 状态筛选，快速定位待办 |
| Schema 视角 | 按 section 归组呈现，贴合抽取 schema |
| 可视化 diff | `diff.json` + 详情内“原始值”对照 |
| 质量控制 | 支持度/置信度/证据数徽章 + 缺证据/冲突显式暴露 |

## 8. 演进路线（数据模型已预留）

- **多人协作 / 权限角色**：`annotation` 文档加 `assignee/reviewer`，服务端加登录与 per-field 锁；`audit_log` 已含 `ts`，补 `actor` 即可。
- **一致性检查**：跨样品同名字段的取值一致性校验（如相同合金成分）。
- **必填校验 / schema 校验**：接入抽取 JSON Schema，做类型与必填红线。
- **PDF 视图**：`source/mineru/*_origin.pdf` 已存在，可加 PDF.js 双视图（当前用对齐更可靠的 cleaned markdown + 图片）。
- **服务端历史版本**：每次 flush 存快照，支持跨会话回滚。

## 9. 技术实现

- **后端**：FastAPI（`app/server.py`）。只读 `extracted/`，写 `annotations/<paper_id>.json`（原子写），导出即时打包。
- **前端**：零构建的原生 ES Module（`app/static/`），三栏工作台 + 客户端状态存储（撤销/重做/自动保存）。
- **证据高亮**：以 evidence block 为单元渲染与高亮，粒度与抽取引用一致，定位精确。
- **启动**：`./run.sh` → http://127.0.0.1:8765
